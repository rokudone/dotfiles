PR レビューコメントを書くときのルール。レビューは相手に気を遣ってするもの。指摘を受けるのは多かれ少なかれストレスであることを前提に書く。目指すのは対等な同僚の文体。指示書でもへりくだりでもない。

## 主観と事実を分ける

- 事実（挙動・行番号・既存実装との比較・仕様との照合）は断定で書く。「〜します」「〜がありません」
- 主観（可読性・命名・構造の良し悪し）は感想として書く。「〜と感じました」「〜が読み取りにくいです」
- 事実を感想めかしてぼかさない。主観を事実めかして断定しない（×「このメソッドは読みにくいです」→ ○「ネストが深くて読みにくいと感じました」）。

## 文体

- 断定命令形の修正指示書スタイル（「〜している。〜する。」）を使わない。
- です/ます調。
- 過剰なクッション言葉（「細かい点ですが」「初見だと迷うかもしれません」「〜していただけると安心です」の乱発）は、かえって馬鹿にした印象を与える。クッションはコメントあたり1つまで。

### 提案形と断定形を分ける

「〜はどうでしょうか」を全ての指摘に機械的に付けない。修正の選択肢があるかどうかで書き分ける。

- **選択肢がある（提案形）**: 設計判断が分かれる、トレードオフがある、相手の意図次第で別解もある。→「〜はどうでしょうか」「〜する方法もあります」
- **やることが一つしかない（断定形）**: 直し方が事実上1通り。未使用コードの削除、既存の対になる実装に揃えるだけ、等。→「〜すれば済みます」「〜が要ると思います」。ここで「どうでしょうか」と聞くと、答えが決まっているのに伺いを立てる空々しい言い回しになる。

判定基準: 「これ以外の直し方があるか？」と自問する。無いなら断定形。「index から未使用の re-export を削る」「テストを既存の手本と同じ形に揃える」は別解が無いので断定形。「カンマ区切り id を query にするか別エンドポイントにするか」は分岐があるので「〜はどうでしょうか」。

### 主体をぼかさない（他人事にしない）

断定形でも「〜することになります」「〜する形になります」のような受け身・無主体の言い回しを使わない。誰がやるのか曖昧で、自然にそうなるかのように聞こえ、指摘が宙に浮く。レビューは「直してほしい」と相手に頼む行為なので、何を望むかを自分の言葉で述べる。

- ×「receipt 側も同じ形に揃えることになります」（他人事・無主体）
- ○「receipt 側も全フィールドを `toEqual` で検証する形にしたいです」
- ○「receipt 側も同じ方式に揃えてほしいです」

「〜が要ると思います」「〜したいです」「〜してほしいです」のように、依頼や要望の主体をはっきりさせる。
- [MUST] 等の priority タグを本文に付けない（指示された場合のみ）。
- 文章自体は prose-style のルールに従う。

## 構成

- 事実（根拠は file:line で）→ 影響 → 提案の順。
- 指摘の動機を保つ。「読みにくいから直したい」という指摘を挙動バグの話にすり替えない。動機が可読性ならそれを冒頭に書く。
- 副次的に直る問題は末尾に一言添えるだけにする。本題にしない。

## リファクタ提案はコードで示す

- 問題の説明だけで終わらせない。直した後の形が伝わるコードブロックを必ず入れる。
- 提案コードに読めばわかるコメントを入れない。許されるのは「# 基底クラス」のような場所ラベルだけ。
- 挙動が変わらないこと（変わる場合はその内容）、既存の override・spec への影響を添える。

## 例

```
call → decode_token → fetch_jwk_set_with_refresh とネストが深くて読みにくいと感じました。   ← 主観は感想として
rescue（91-93行目）が初回 fetch の通信失敗まで拾って再 fetch するため、外部 IO が 2 回走ります。 ← 事実は断定で
call 直下に「鍵を選ぶ → decode → クレーム検証」を並べることを提案します。              ← 提案

（直した後の形のコードブロック）

spec は call 経由への書き直しが必要です。副次的に二重 fetch もなくなります。          ← 影響と副次効果は末尾に
```

## 投稿フロー

投稿前に最終文面を一字一句提示してユーザーの承認を得る。承認後にタグ付けや注記の取り込み等の加筆・変形をしない。修正したら再提示する。
