## 3. 技術理解度マップ

## 4. プロンプティング力の評価

### 強み

- **誤りを即座に指摘する批判的フィードバック**: AIの回答に対して「そうじゃなくて」「違う」「なんで？」と迷わず反論し、正確な意図を伝え直している。「だから、今どれがキーバインドに割当たってるか聞いてるの！！！！！！！！！！」のように強い語気で優先事項を明示できている（Claude Code）
- **段階的な作業分割の指示**: 「1. まずは正確にこれを列挙してください 一言一句変わらず、順番も変わらず」「2. こちらについても検討しなさい。コードを読みなさい チームを組むこと」のように手順を番号で明示している（Claude Code）
- **コンテキストの積極的な提供**: Jira URL・Confluence URL・コードの貼り付けを適切に使ってAIに情報を渡している。「@今のところ集まってる情報.md とかに当初の想定は入ってるので」などファイル参照指定も的確（Claude Code）
- **システムの制約への配慮**: 「casy-api は4並列なので精々12秒くらいか、やらんほうがいいな」「rspec は単一ファイル実行しかできない制約があります」など実行環境の制約を自発的にAIに伝えている（Claude Code）
- **Claude Code 固有機能の活用**: `/casy-step3-review`（46回）・`/casy-step4-fix`（30回）・`/rate-limit-options`（23回）・`/casy-work`（15回）など CCK スキルをワークフローに組み込み、プロセスを自動化している（Claude Code）

### 改善ポイント

- **曖昧な量的指示**: 「なんかさ、直しても直しても一生出てくるのってなんで？」「未対応のもの結構存在しない？」のような感情的な問いかけが複数見られる。→ **改善提案**: 「未対応の MUST 項目を全件列挙してください」のように定量的・具体的な依頼に変換する
- **前提の後出し**: 「いや、そうじゃなくて、PATHSをちゃんと組み立てるべきだって言ってる」「そうじゃなくて、別プロセスで起動してるっぽい感じがするんだよな」のように、AIが前提を誤解した後に正しい前提を提示するケースが散見される。→ **改善提案**: 「〜という前提で」「〜の観点から」を最初の指示に含めておく
- **タイポ・誤字を含む指示**: 「comit push pr」「finging-common」「生合成担保」「洋画あります」など、タイポが一定頻度で見られる。重要なパラメータ（ブランチ名・APIパラメータ名など）に誤字が混入するとバグの原因になる（Claude Code）

### 特徴的な癖

- 短い承認・拒否を番号で大量処理する。「3 何言ってるかわからん / 4 y / 5 n / 6 y / 7 y / 8 n」のように、レビュー項目を1行で大量処理する方式は独特で効率的（Claude Code）
- 「なんで〜なん？」「〜じゃね？」という口語的な問いかけでコードの問題を指摘する。命令形よりもソクラテス式問答に近いスタイル（Claude Code）
- 「チーム組んで」を並列処理の起動トリガーとして使う。要件リストの末尾に「チーム組んで」と付け加えることでマルチエージェント実行を開始するパターンが定着している（Claude Code）
- `commit` / `comit push pr` / `update して push` など、1〜3語の git 操作ショートカットを頻用する（Claude Code）

## 5. AI活用スタイル

### 主体的に活用できている領域

- **ワークフロー設計・ツール構築**: CCK のエージェント構成・スキル設計・プロセス数チューニングなど、AI ツールそのものの設計をAIと協働で進めている。「review-1に claude / codex のモードがある方が自然かなあ」「review-agent-0 → review-0-~~.md でどのレビュー化がわかるようになってて欲しい」と具体的なUX方針を提示している（Claude Code）
- **コードレビュー・品質管理**: AIが出力したレビュー結果を自分で評価し、採否を決定している。「MUST の1は casy の仕様を把握していないレビューだわ」「BUG-4 はあれだわ、スポットデフォルトが基本だわ 無視して」のように独自判断でAI意見を上書きできている（Claude Code）
- **要件定義・設計議論**: Jira/Confluence のチケット・ページを参照しながら「今の実装って、多分要求定義を満たさないと思うんだよね」と主体的に仕様差異を指摘し、設計の方向性を決定している（Claude Code）

### AIへの依存が高い領域

- **実装コードの生成全般**: 機能設計後の実装コード生成はほぼ全面的にAIに委任している傾向がある。step2（実装）はスキルを起動して放置するパターンが定着している可能性がある。→ **自立に向けた提案**: 生成されたコードの核心部分（ビジネスロジック）だけは自分でも目を通し、仕様意図との乖離がないかをチェックするルーティンを設けると品質リスクを低減できる
- **エラーのデバッグ調査**: 「これなんで？」「これなんだ？」とエラーや想定外の挙動をそのままAIに投げるケースが多い。原因仮説を自分で立てた上で「〜が原因ではないか確認して」と投げる形に切り替えると、AIの的外れな回答を減らせると推測される（Claude Code）

### ツール別の活用傾向

- **Claude Code**: ワークフロー全体（設計→実装→レビュー→修正→E2E→PR）をスキルで自動化しており、CCK の旗振り役として Claude Code を中枢に置いている。スキル自体の開発・改善にも Claude Code を使っており、二重に活用している（Claude Code）
